# РАГ, помогающий с учебой и подготовкой к ЕГЭ

## 1. Цели и предпосылки

### 1.1. Актуальность продукта

* Бизнес-цель:  
Создать AI-помощника для подготовки к Единому Государственному Экзамену, который повысит эффективность самостоятельных занятий, снизит уровень тревожности перед экзаменами, а также обеспечит учащихся доступом к структурированной, достоверной информации в любой момент.
* Что будем считать успехом с точки зрения бизнеса:
  * Функциональный MVP, способный одновременно обслуживать не менее 10 сессий с сохранением контекста диалога для каждого, который дает полезные ответы на основе загруженной базы знаний со средним временем ответа не более 10 секунд.
  * Рост вовлечённости: не менее 70% пользователей завершают свою первую сессию (задают более 3 вопросов) и возвращаются для повторного обращения в течение недели.
  * Позитивный пользовательский feedback: средняя оценка сессии ≥ 4.0/5 по метрикам: "полезность ответа", "ясность объяснений", "доверие к источнику".
  * Если ответа нет в загруженных материалах, система должна честно сообщить об этом, а не генерировать "правдоподобный" ответ (предотвращение галлюцинаций).
  * Подтверждение технологической эффективности: должен обеспечивать точность ответа ≥ 80% в соответстивии с загруженными материалами (без галлюцинаций).
  * Подтверждение безопасности: модель должна вежливо отказываться отвечать на вопросы, не связанные с учебой, а также указывать, что является простым помощником, а не профессиональным консультантом. Модель не должна предоставлять неверных, опасных или запрещенных рекомендаций.
 
### 1.2. Бизнес-требования

* Требования:
    * Система должна вести многошаговый диалог, запоминая историю вопросов и ответов в рамках одной сессии.
    * Использование RAG для получения акутальных учебных материалов.
    * Возможность интеграции в веб-интерфейс или Telegram-бота для удобства пользователей.

* Ограничения:
    * Модель не заменяет профессионального репетитора и не даёт рекомендаций, выходящих за рамки учебных материалов.
    * Ответы генерируются исключительно на основе загруженной базы знаний, а при отсутствии информации система должна честно сообщать об этом.
    * Ответы должны быть консервативными, нейтральными и строго соответствовать учебной программе, без субъективных ответов и оценок.
    * Время ответа ограничено 10 секундами для поддержания приемлемого пользовательского опыта.
    * Система не хранит персональные данные пользователей.
 
* Что будем считать успешным пилотом? Система способна провести сессию длительностью 10–20 минут, сохраняя контекст диалога и не теряя нить разговора. Ответы модели должны быть безопасными, соответствовать учебной программе и направлять пользователя к решению задачи с помощью наводящих подсказок, а не просто выдавать готовый ответ.

### 1.3. Предпосылки решения

* Почему мы решили создать этого AI-помощника:
  
    * Высокий уровень стресса у школьников перед экзаменами.
    * Разрозненность и неструктурированность информации.
    * Нехватка персонализованного подхода в школах
    * RAG позволяет строить решения, которые дают ответы на основе достоверных источников, что критично важно для образованиями.

### 1.4. Постановка задачи

Разработать LLM-агента для подготовки к ЕГЭ, который ведёт корректный диалог с учеником, опираясь на загруженные учебные материалы и официальные задания ФИПИ, и предоставляет персонализированные, безопасные и достоверные ответы в формате диалога. В перспективе агент должен поддерживать интеграцию с внешними API для расширения базы знаний.

## 1.6. Что входит в MVP, а что не входит

Входит:
* RAG-ядро: индексация учебных материалов и быстрый поиск по ним.
* LLM-агент: диалоговая логика для ответов на вопросы по ЕГЭ.
* Telegram-бот для взаимодействия с пользователем.

Не входит:
* Авторизация и хранение персональных данных.
* Мобильное приложение.
* Распознавание формул и изображений.

## 2. Подготовка данных

### 2.1. Описание данных и источники

Для работы RAG-системы я формирую собственный датасет, который будет использоваться для векторного поиска. Данные будем брать из:
1. Официальные материалыы ФИПИ: кодификаторы, демоверсии, открытые банки заданий ЕГЭ.
2. Учебные пособия и методички, рекомендованные Министерством просвещения в формате PDF/DOCX.
3. Статьи и методические рекомендации с объяснением алгоритмов решения сложных задач.

Все материалы собираются вручную из открытых, проверенных источников. На этапе MVP планирую загрузить ~10 документов.

#### 2.1.1. Формат и структура хранения

Данные подаются в виде очищенных PDF и DOCX. Разбиение выполняется рекурсивным сплиттером на чанки по 512 токенов (как оптимальный баланс между сохранением контекста и скоростью поиска) с перекрытием 50-100 токенов. К каждому чанку прикрепляются метаданные: источник, страница, тип контента (теория/задание/решение). После векторизации чанки сохраняются в FAISS для быстрого поиска.

#### 2.1.2. Критерии отбора источников

Чтобы гарантировать качество и достоверность базы знаний, я отбираю материалы по следующим критериям:

* Официальность: приоритет отдаётся документам ФИПИ и учебникам из федерального перечня Министерства просвещения.
* Актуальность: все источники должны быть не старше 2021 года, чтобы соответствовать текущей спецификации ЕГЭ.
* Читаемость: я выбираю только текстовые файлы без сканов, либо те, которые можно качественно распознать через OCR.
* Структурированность: предпочтение отдаётся материалам с чёткой разбивкой на темы, главы и параграфы — это упрощает чанкинг и сохраняет связность контекста в ответах системы.

## 3. Архитектура решения

### 3.1. Технологический стек

Для реализации RAG-системы я планирую использовать следующий стек:

*   Бэкенд: FastAPI/Flask — для создания API-эндпоинтов и обработки запросов.
*   Векторная БД: ChromaDB — для локального хранения эмбеддингов и быстрого поиска релевантных чанков.
*   LLM: OpenAI GPT-4o mini (на рассмотрении, окончательный выбор после тестирования скорости и качества) — для генерации ответов.
*   Эмбеддинги: OpenAI text-embedding-3-small (аналогично, изучаю) — для преобразования текста в векторы.
*   RAG-фреймворк: LangChain — для сборки пайплайна.
*   Интерфейс: Telegram Bot API — как наиболее быстрый способ предоставить доступ пользователям.

## 4. Аналитика и метрики

Чтобы понимать, как работает система, будем собирать следующие метрики:

### 4.1. Метрики качества ответа

* Точность ответа проверяется вручную на выборке из 10 случайных диалогов путём сравнения ответов системы с исходными материалами из базы знаний (целевое значение — не менее 80%). 
* Время ответа измеряется автоматически как интервал между запросом пользователя и получением финального ответа от LLM (порог — не более 10 секунд). 
* Доля полезных ответов собирается через опрос после сессии с вопросом «Был ли ответ полезен?» (варианты: Да / Нет / Частично), целевое значение — не менее 70% положительных ответов. 
* Доля отказов (когда система не нашла информацию в базе знаний) должна составлять не более 20% от всех запросов.

### 4.2. Метрики вовлеченности пользователей

* Повторные визиты — процент пользователей, вернувшихся в систему в течение 7 дней после первого обращения (цель — не менее 40%). 
* Средняя длина сессии оценивается по количеству вопросов и времени между первым и последним сообщением (желаемый диапазон — 10-15 минут). 
* Оценка пользователя собирается после сессии по шкале от 1 до 5 по трём критериям: полезность, ясность объяснений и доверие к источнику (средняя оценка ≥ 4.0).

### 4.3. Оценка безопасности

* Система тестируется на 10 вопросах, не связанных с учебой. Цель — 100% вежливых отказов с объяснением, что система создана для помощи в учёбе. 
* Также проверяется, что модель не даёт опасных или запрещённых рекомендаций (например, не советует препараты или опасные действия).
* Тестирование на промпт-инъекции и попытки выйти за рамки роли
